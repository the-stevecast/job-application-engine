# Output Rules

> **How the agent uses this file:** Governs formatting decisions for resumes and cover letters. These rules apply to every application generated. They are the same for all users and do not need to be personalized.

---

## Resume Format Rules

### Design for 6-10 Second Scanning

Hiring managers skim. Every section must earn its space. Front-load the strongest signal in every section: summary, competencies, and bullet order.

### Summary

- 4-5 lines max. No single-block paragraphs.
- Structure: who I am + domain + recent focus + key capability + differentiator.
- Must be specific enough that a reader knows in 5 seconds what the person does.
- Mirror the job's language without copying.

### Core Competencies

- 6-8 items max. Single column. No subcategories.
- Reorder to match the job's priorities (most relevant first).
- ATS-safe: plain text, no special formatting, no tables.

### Bullet Format (all roles)

Every bullet must follow: **Ownership + System/Process + Impact**

- Start with a strong ownership verb (Built, Designed, Led, Established, Launched, Defined, Created, Architected, Grew, Reduced, Restructured).
- State what was built, designed, or implemented.
- End with the operational result (faster, reduced, eliminated, improved, enabled, saved, grew).
- 1-2 lines max per bullet. If it runs to 3 lines, split it or cut words.
- No soft verbs as openers (Helped, Supported, Assisted, Participated, Worked on).
- No bullets that are purely descriptive with no result.

### Bullet Counts by Role

- Primary/most recent role: 5-7 bullets (heaviest section)
- Secondary roles: 3-4 bullets each
- Earlier roles: 1-2 bullets or compress to a single descriptive line
- Education: Always include

### Timeline Accuracy

- Do not imply capabilities started earlier or later than they actually did.
- When a user's context specifies when certain skills or roles began, reflect that accurately.
- If a capability evolved over time (e.g., a skill started in one role and deepened in another), note the timeline correctly.

### ATS Safety

- Single column layout only. No tables, no multi-column formats.
- Standard section headings: Professional Summary, Core Competencies, Professional Experience, Education.
- No headers/footers, graphics, icons, or text boxes.
- Use standard bullet characters (-, *).

### Company Acquisition Formatting

If a prior employer was acquired, format the heading to show the chain:
`Company Name (acquired by Acquirer, now Current Entity)` — This communicates credibility and scale. Apply when acquisition context is present in the bullet bank.

## Cover Letter Format Rules

### Structure

4 paragraphs maximum. Each paragraph 3-5 sentences.

- **Paragraph 1 (Capability Lead):** Open with what the candidate brings, not praise of the company. State the core capability that maps to this role.
- **Paragraph 2 (Operational Example):** One concrete example: specific situation, what was built or done, operational result.
- **Paragraph 3 (Differentiator):** What makes this candidate's approach distinctive. Connect to this role's specific needs.
- **Paragraph 4 (Close):** Direct, confident. Express interest in a conversation.

### Tone

- Tone is set by `context/writing-voice.md`. Follow it.
- Default if voice is not specified: peer-to-peer, confident, not pitching.
- Never open with: "I'm excited about...", "I was thrilled to see...", "Your company's mission resonates with me..."
- No desperation, no "I would be honored."

### Style

- Follow all rules in `context/writing-voice.md` — including the avoid list and punctuation preferences.
- No buzzwords or hype language.
- Every claim should be backed by evidence somewhere in the letter or resume.

---

## General Output Rules

- Concrete and execution-focused. Avoid vague phrases ("helped", "supported") unless paired with clear ownership.
- Plain language. Avoid jargon for its own sake.
- If metrics are missing, use [TBD] as a placeholder — never invent numbers.
- Every output is specific to this role and this person. Nothing generic.
